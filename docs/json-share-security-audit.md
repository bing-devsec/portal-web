| 项目 | 内容 |

| 审计范围 | 服务端 `server/api/share-json.ts`、`server/api/keys.ts`；前端 `components/JsonTool/ShareDialog.vue`、`components/JsonTool/JsonTool.client.vue`；指纹链路 `utils/fingerprint.ts` |

## 1. 概述

JSON 分享功能允许匿名用户把一段 JSON 数据上传到服务器生成短链接，供他人通过链接（可选密码）访问。本报告对该功能进行深度安全评估，覆盖身份模型、访问控制、拒绝服务、数据存储、传输安全、并发安全等维度，给出风险分级与修复建议。

### 1.1 架构与攻击面

- **入口**：`/api/share-json`，单一 handler 处理三种方法：
  - `GET`：读取分享 / 列出"我的分享"（`?mine=1`）
  - `POST`：创建分享
  - `DELETE`：删除分享
- **身份模型**：浏览器用**公开公钥**加密一个 32-hex 指纹（FingerprintJS 产物）→ 请求头 `x-client-id` → 后端解密得明文指纹，作为唯一"用户身份"，用于配额、限流、列表过滤、删除鉴权。
- **存储**：明文 JSON 文件存于 `/tmp/shares/{id}.json`，配合 `index.json` 索引；内存中维护索引缓存（30s TTL）、限流计数、到期删除定时器。

### 1.2 风险汇总

| 编号 | 风险 | 等级 |
|------|------|------|
| H1 | 身份模型可被任意伪造（配额/限流/越权根源） | 严重 |
| H2 | 全局存储无上限 → 磁盘耗尽 DoS | 严重 |
| H3 | 受密码保护分享可被暴力破解 | 严重 |
| M1 | 索引读改写竞态，数据/配额错乱 | 中 |
| M2 | 密码以明文存储 | 中 |
| M3 | 密码经 URL 查询串传输与回显 | 中 |
| M4 | 无 CSRF/来源校验，且 GET 带写副作用 | 中 |
| M5 | 指纹解密接口扩大内网攻击面 | 中 |
| L1 | Host 头注入影响分享链接 | 低 |
| L2 | 错误信息可探测分享是否存在 | 低 |
| L3 | 限流为单进程内存态 | 低 |
| L4 | 顶层 catch 直接回显内部错误 | 低 |
| L5 | 内容侧 XSS / 路径遍历（已规避，确认安全） | 信息 |

## 2. 严重风险（High）

### H1. 身份模型可被任意伪造 —— 配额/限流/越权的总根源

**位置**：`server/api/share-json.ts:595-602`（指纹获取）、`:464`（我的分享过滤）、`:805`（删除鉴权）

**描述**：
功能的"用户身份"完全建立在一个**客户端自选、用公钥加密**的指纹之上。公钥通过 `/api/keys` 公开可取（`utils/fingerprint.ts:70` 即证明任意客户端都能获取）。因此攻击者可以用公钥加密**任意** 32-hex 字符串当作指纹。

**影响**（连锁）：
- **配额绕过**：每换一个随机指纹即视作"新用户"，`MAX_SHARES_PER_FINGERPRINT=5`、`MAX_SIZE_PER_FINGERPRINT=10MB`、`MAX_USERS=10000`（`CONFIG`，`:71-82`）全部失效。
- **限流绕过**：`checkRateLimit`（`:296-311`）按指纹计数，换指纹即重置。
- **越权删除/列举**：`DELETE` 鉴权（`:805`）与"我的分享"（`:464`）均以指纹为准。指纹来自浏览器可观测特征，**不是机密**；若攻击者复现受害者指纹（同环境碰撞或社工获取），即可冒充其删除/列举分享。

**根因**：把"匿名限流提示信号"误用为"身份鉴权凭证"。指纹只能作为弱去重信号，不能用于授权（authz）。

**建议**：
- 明确指纹仅作弱去重，不作为安全边界；
- 真正的滥用防护改用 **IP 维度限流 + 全局配额**（见 H2/H3）兜底；
- 越权敏感操作（删除）不应仅凭指纹，需结合密码或一次性凭证。

---

### H2. 全局存储无上限 → 磁盘耗尽 DoS

**位置**：`server/api/share-json.ts:699-707`（仅单指纹限额）、`:746-757`（totalSize 只增不校验）

**描述**：
代码只限制"单指纹 10MB"，`index.totalSize` 持续累加但**从未对全局总量做上限校验**。结合 H1（指纹可无限伪造），攻击者可：无限造新指纹 × 每指纹 5 份 × 每份 2MB → **无上限写盘**。

**影响**：撑爆 `/tmp` 直至容器/主机不可用；`/tmp` 写满可能波及同机其它服务。

**建议**：
- 增加**全局存储上限**与**全局分享数上限**，超限拒绝创建；
- 存储目录独立挂载并设配额（quota），避免影响宿主；
- 结合 IP 限流降低构造速率。

---

### H3. 受密码保护分享可被暴力破解

**位置**：`server/api/share-json.ts:642-648`（限流仅在 POST）、`:542-551`（密码比较）、`:578-593`（密码规则）

**描述**：
- GET 读取路径**无任何限流**（限流只加在 POST）。
- 密码校验为 `shareData.password !== password`（`:544`），**非恒定时间比较**，且无失败计数、无锁定。
- 密码**无最小长度**要求，可设置极短密码。

**影响**：攻击者拿到 shareId 后可对密码不限速暴力枚举，短密码可被快速破解。

**建议**：
- 对 GET（尤其携带密码的读取）增加**限流 + 失败锁定**；
- 使用**恒定时间比较**（如 `crypto.timingSafeEqual`）；
- 设定密码最小长度与复杂度。

## 3. 中风险（Medium）

### M1. 索引读改写竞态，可致数据/配额错乱

**位置**：`server/api/share-json.ts:746-757`（创建）、`:202-208`（writeIndex）、`:122-125`（缓存）

**描述**：`index.json` 的 read-modify-write（创建、删除、过期清理）**无任何锁/串行化**，且有 30s 内存缓存。并发请求会相互覆盖。`fs.writeFile` 非原子，进程崩溃可损坏整个索引。

**影响**：配额计数不准（可绕过 5 份限制）、`totalSize` 漂移、条目丢失、索引文件损坏。

**建议**：对索引写入串行化（单写队列/文件锁）；采用原子写（写临时文件后 `rename`）。

---

### M2. 密码以明文存储

**位置**：`server/api/share-json.ts:733-742`

**描述**：密码原样写入分享文件。文件虽为 0600 权限，但任何能读取磁盘/备份/快照者都能看到所有分享密码。

**建议**：存储密码哈希（bcrypt/scrypt/argon2），校验时比对哈希。

---

### M3. 密码经 URL 查询串传输与回显

**位置**：`components/JsonTool/JsonTool.client.vue:7562-7582`（GET ?password=）、`:7656-7658`（回写地址栏）

**描述**：
- 读取走 GET `?password=`，密码会进入服务器访问日志、nginx/代理日志、浏览器历史、Referer。
- 密码输错后又将其 `set('password', value)` 写回当前 URL，进一步在历史/地址栏留存。

**建议**：密码改走 POST body 传输；不要把密码回写到地址栏。

---

### M4. 无 CSRF/来源校验，且 GET 带写副作用

**位置**：`server/api/share-json.ts:804-812`（DELETE 密码分支）、`:553-556`（GET 写 accessCount）

**描述**：
- POST/DELETE 无 CSRF token、无 Origin/Referer 校验。DELETE 的"密码删除分支"不依赖自定义头，跨站可构造请求。
- **GET 带写副作用**：每次读取都 `writeShareData` 更新 accessCount，使"读"可被滥用为高频写盘（放大 DoS），且违反 GET 幂等无副作用原则。

**建议**：对状态变更请求做 Origin/Referer 校验或 CSRF token；将 accessCount 更新改为异步/节流，或与读取解耦。

---

### M5. 指纹解密接口扩大内网攻击面

**位置**：`server/api/share-json.ts:14-50`（调用 `POST {ssrApiBase}/user/fingerprint/decrypt`）

**描述**：改造后 Nuxt 将指纹解密委托给 Go 后端。需确保该接口仅内网可达、自身有限流，否则可能成为 **RSA 解密预言机**，被高频调用做计时/资源类攻击。

**建议**：限制该接口仅内网访问；增加限流；失败响应不回显敏感信息。详见 `docs/fingerprint-decrypt-backend-spec.md`。

## 4. 低风险与加固项（Low / Hardening）

### L1. Host 头注入影响分享链接
**位置**：`server/api/share-json.ts:762-768`、`:487`
`shareUrl` 用 `getRequestURL(event).origin` 拼接。若上游未固定 Host，攻击者改 Host 头即可让返回链接指向钓鱼域名。**建议**：使用配置中的固定 `siteUrl`。

### L2. 错误信息可探测分享是否存在
**位置**：`server/api/share-json.ts:519-551`
"不存在/已过期"与"密码不正确"消息不同，可被探测 shareId 是否存在。鉴于 ID 为 128bit 随机、枚举不可行，影响有限。**建议**：统一模糊化错误消息。

### L3. 限流为单进程内存态
**位置**：`server/api/share-json.ts:128`
`fingerprintRateLimit` 为进程内 Map，多实例/重启即失效，不适合水平扩展。**建议**：迁移至 Redis 等共享存储。

### L4. 顶层 catch 直接回显内部错误
**位置**：`server/api/share-json.ts:841-845`
直接返回 `error.message`，可能泄露内部细节。**建议**：对外返回通用错误，内部记录详细日志。

### L5. 内容侧 XSS / 路径遍历（已规避，确认安全）
- 分享的 `jsonData` 仅通过 Monaco `inputEditor.setValue()`（`JsonTool.client.vue:7600`）作为**纯文本**载入，未走 `v-html`，**无存储型 XSS**。
- 路径遍历已通过 `isValidShareId`（`:134-140`）+ `getShareFilePath`（`:143-163`）双重校验拦截。
- 文件写入使用 0600 权限（`:277-280`）。
- **结论**：这几处实现良好，无需修改。

## 5. 修复优先级建议

| 优先级 | 问题 | 建议方向 |
|--------|------|----------|
| P0 | H2 磁盘耗尽 | 增加全局存储上限与全局分享数上限，超限拒绝创建 |
| P0 | H3 密码爆破 | GET（尤其带密码）加限流 + 失败锁定；恒定时间比较；密码最小长度 |
| P0 | H1 身份伪造 | 明确指纹仅作弱去重；P0 风险靠全局配额 + IP 限流兜底，而非依赖指纹 |
| P1 | M1 索引竞态 | 写入串行化/加锁；原子写（临时文件 + rename） |
| P1 | M2 密码明文 | 改存哈希 |
| P1 | M3 密码走 URL | 改 POST body 传输，不回写地址栏 |
| P2 | M4 | 状态变更请求做 CSRF/来源校验；GET 去副作用 |
| P2 | M5 | 指纹解密接口限内网 + 限流 |
| P3 | L1–L4 | 固定域名、收敛错误信息、限流迁共享存储、收敛错误回显 |

## 6. 附录：已确认安全的实现

为避免误改，以下实现经审计确认**当前是安全的**，列出供参考：

- **分享 ID 生成**：`randomBytes(16)`（128 bit）→ 不可枚举（`:426-428`）。
- **路径遍历防护**：ID 正则校验 + 规范化路径前缀比较（`:134-163`）。
- **文件权限**：分享数据文件 0600（`:277-280`）。
- **内容渲染**：纯文本载入编辑器，无 HTML 注入面（`JsonTool.client.vue:7600`）。
- **JSON 校验**：创建与加载均做 `JSON.parse` 校验与大小限制。
